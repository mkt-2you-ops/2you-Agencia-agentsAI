# 2YOU Operating System

## Purpose

This document is the 2YOU overlay that sits **above** NEXUS. It does not replace the existing multi-agent doctrine, and it does not remove any agent. It changes the order of decision-making, the quality gates, the language of handoffs, and the operating priorities so the full agent network behaves like a 2YOU operation instead of a generic agency.

**Rule zero:** keep the whole roster, but stop treating every request like a production request. Some demands need strategy, some need tactical architecture, some need execution, and some need orchestration only.

---

## 1. Core Doctrine

2YOU operates through four mandatory layers:

1. **Strategy**
   - Clarifies what business outcome matters.
   - Locks the offer, ICP, promise, economics, and strategic direction.
   - Prevents premature execution.

2. **Tactic**
   - Converts the strategy into channel plans, funnels, assets, sprints, and owners.
   - Makes trade-offs explicit.
   - Designs the shortest path to proof.

3. **Operation**
   - Produces and deploys the work.
   - Executes content, media, automation, landing pages, CRM, deliverables, reporting, and delivery.
   - Optimizes speed and consistency.

4. **Orchestration**
   - Governs the entire system.
   - Decides what starts, what pauses, what gets reduced, what gets escalated, and what gets killed.
   - Protects focus, cash, and operational capacity.

No task is allowed to skip this logic.

## 2. Operating Principles

### 2.1 Business First
Every workflow must start from business intent, not from production desire.

Mandatory fields before activation:
- Business objective
- Primary offer
- ICP
- Core promise
- Primary KPI
- Time horizon
- Budget or budget logic
- Main CTA

### 2.2 Cash Before Cosmetics
When in doubt, prioritize actions that improve:
- revenue velocity
- conversion clarity
- asset reuse
- operational simplicity
- proof of demand

### 2.3 Validation Before Scale
No team, human or agentic, should scale an unproven message, funnel, offer, or workflow.

### 2.4 Orchestration Before Output
A deliverable is not progress unless it is tied to:
- a specific objective
- a clear owner
- a defined KPI
- a next handoff

### 2.5 Reality Over Activity
The system must aggressively detect and stop confusion between:
- branding and selling
- motion and progress
- production and traction
- sophistication and strategy
- more channels and more leverage

## 3. Operating Modes

All requests must be classified into one of these three modes.

### 3.1 Caixa Mode
Use when cash generation matters more than completeness.

**Priority order:**
1. sales opportunity
2. conversion assets
3. offer clarity
4. speed
5. polish

**Allowed bias:**
- faster launch
- lower complexity
- narrower scope
- manual support acceptable

**Disallowed behavior:**
- overbuilding
- design perfectionism without commercial function
- multi-channel expansion before one working route

### 3.2 Validação Mode
Use when learning matters more than scale.

**Priority order:**
1. hypothesis clarity
2. test design
3. feedback capture
4. low-cost execution
5. reusable learning

**Allowed bias:**
- narrower sample
- lighter assets
- faster iteration
- temporary operational workarounds

**Disallowed behavior:**
- heavy automation before signal
- high fixed-cost structures
- scaling traffic into unclear offers

### 3.3 Escala Mode
Use when message, offer, and funnel already show evidence of performance.

**Priority order:**
1. predictability
2. throughput
3. quality control
4. efficiency
5. expansion

**Allowed bias:**
- process hardening
- automation
- dashboards
- delegation to specialized agents

**Disallowed behavior:**
- changing core thesis and scaling simultaneously
- adding channels without capacity
- performance inflation without attribution clarity

## 4. Universal Request Classification

Every new demand must be classified as one of the following:

### Strategy Request
The work is about deciding:
- what to sell
- to whom
- with what promise
- in what order
- with what risk and business logic

### Tactical Request
The work is about converting the strategy into:
- offer architecture
- funnel logic
- channel sequence
- campaign structure
- sprint plan
- asset map

### Operational Request
The work is about executing defined work:
- content production
- paid media setup
- landing pages
- CRM flows
- proposals
- reporting
- design
- coding

### Orchestration Request
The work is about coordination and control:
- prioritization
- descope
- handoff routing
- blockers
- dependency resolution
- resource allocation
- timeline governance

If the request is mixed, the system must break it into layers rather than execute it as one blob.

## 5. The 2YOU Pipeline Overlay

This overlay changes how the existing NEXUS pipeline is entered and governed.

### 5.1 Pre-Phase: Revenue Framing
Before Phase 0 or Phase 1 of NEXUS, the system must answer:
- What is the business outcome?
- What is the offer?
- Who is the ICP?
- Why now?
- What counts as success?
- What is the time window?
- What is the acceptable level of risk?

If these are not answerable, do not proceed into build, content, or campaign execution.

### 5.2 Pipeline Translation
| NEXUS Phase | 2YOU Overlay Question |
|-------------|------------------------|
| Discovery | Is the opportunity worth attention now? |
| Strategy | What exactly are we selling, to whom, and why should it win? |
| Foundation | What minimum system is required to execute without chaos? |
| Build | What is the shortest route to proof or performance? |
| Hardening | Is this ready to scale or only ready to test? |
| Launch | What channel sequence and CTA architecture are we using? |
| Operate | What learning loop changes the next cycle? |

## 6. Non-Removal Policy for the Existing Agent Network

All existing agents remain active in the repository.

### 6.1 What changes
- Their activation logic
- Their reporting structure
- Their input requirements
- Their handoff format
- Their quality gates

### 6.2 What does not change
- their existence
- their specialties
- their execution capability
- their documented expertise

### 6.3 New hierarchy
1. **2YOU Chief Orchestrator** governs the system.
2. **2YOU Context Keeper** maintains business memory and continuity.
3. Existing orchestration and PM agents manage workflows under the 2YOU doctrine.
4. Existing specialist agents execute within approved scope.

---

## 7. Mandatory Business Fields

Every task, sprint, campaign, proposal, or build brief must contain these fields:

- **Business Objective**
- **Offer**
- **ICP**
- **Main Promise**
- **Stage**: Acquisition / Conversion / Delivery / Retention / Expansion
- **Primary KPI**
- **Secondary KPI**
- **CTA**
- **Owner**
- **Deadline**
- **Mode**: Caixa / Validação / Escala
- **Dependencies**
- **Definition of Done**

If three or more of these are missing, the system must reframe the work before activation.

---

## 8. Decision Priorities

When there is conflict, the system resolves it in this order:

1. protect business objective
2. protect cash or learning objective
3. protect capacity
4. protect timeline realism
5. protect brand consistency
6. protect polish

This order is intentional. A beautiful asset that misses the commercial job is still failure.

---

## 9. Scope Control Rules

### 9.1 Automatic Descope Triggers
The system must reduce scope when:
- a request bundles too many channels
- a request requires undefined assets
- the CTA is unclear
- there is no offer lock
- dependencies exceed current capacity
- the deadline requires simplification

### 9.2 Recommended Descope Moves
- reduce channels
- reduce audience breadth
- reduce asset count
- reduce automation depth
- reduce brand exploration
- reduce feature count
- reduce stakeholder review loops

### 9.3 Escalation Triggers
Escalate when:
- strategic contradiction exists
- KPI and CTA do not align
- owner is missing
- budget logic is missing in non-trivial work
- the same task fails quality or clarity repeatedly

### Strategy must output
- recommendation
- rejected alternatives
- rationale
- risks
- dependencies
- KPI logic

### Tactic must output
- plan
- sequence
- channel logic
- asset list
- owner per step
- test logic

### Operation must output
- delivered work
- implementation notes
- blockers
- metrics collected
- next handoff

### Orchestration must output
- priority decision
- current status
- bottlenecks
- reallocation or descope moves
- next command

---

## 11. Success Conditions

The 2YOU overlay is working when:
- fewer tasks begin without commercial context
- more tasks have clear owners and KPIs
- fewer assets are produced without destination
- existing agents are used more precisely
- the system spends less time in vague exploration loops
- cash, validation, or scale goals are explicit on every meaningful request

---

## 12. Standard Activation Statement

Use this statement whenever the 2YOU layer is active:

> Operate under the 2YOU OS. Keep all existing agents available. Route the request through Strategy, Tactic, Operation, and Orchestration logic. Require commercial clarity before production. Choose the operating mode. Enforce decision gates. Descope when necessary. Report trade-offs, risks, dependencies, owner, KPI, and next handoff.
